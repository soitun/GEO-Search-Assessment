## 五、改进建议（按优先级）

### P0 — 立即行动

#### P0-1. SPA页面实现SSR或静态预渲染
- **影响问题**: Q4（活动页面）、Q5（贡献页面）
- **影响平台**: 全部
- **现状**: mindspore.cn活动页面和贡献页面均为Nuxt.js SPA，AI搜索引擎和爬虫无法索引内容
- **具体措施**:
  1. 将 `/activities` 和 `/contribution` 页面改为SSR（服务端渲染）
  2. 或提供静态预渲染版本（prerender）
  3. 添加 `/sitemap.xml` 包含所有关键页面
  4. 添加 `<meta name="robots">` 和 OpenGraph 标签
- **预期效果**: 让AI平台能索引核心社区页面，消除Q4/Q5信息空白

#### P0-2. 发布官方版本发布策略文档
- **影响问题**: Q2（版本发布节奏）
- **影响平台**: 全部（所有平台都在猜测）
- **现状**: 无官方版本发布策略文档，导致平台各说各话（月度/双月/半年），DeepSeek甚至编造了"奇偶版本策略"
- **具体措施**:
  1. 创建 `/releases` 或 `/version-policy` 页面
  2. 内容：发布频率承诺、LTS政策、维护周期、版本命名规则
  3. 在 GitHub 上正式使用 Releases 功能（当前releases页面为空，只有tags）
  4. 在 GitHub README 中添加仓库迁移说明（GitHub仅到v2.3.0）
- **预期效果**: 消除平台对发布节奏的猜测和编造

#### P0-3. 强化FAQ中"不支持直接读取其他框架模型"的表述
- **影响问题**: Q9（模型格式支持）× 3平台
- **影响平台**: ChatGPT、豆包、DeepSeek（3/4平台给出严重错误回答）
- **现状**: 当前FAQ表述力度不够，3/4平台将MindSpore描述为"支持直接读取PyTorch/TF模型"
- **具体措施**:
  1. 在FAQ中用**加粗/告警框**格式强调："MindSpore **不支持** 直接读取其他框架模型"
  2. 列出明确的"不支持"操作清单
  3. 明确区分：主框架原生能力（仅MINDIR）vs MindSpore Lite转换工具能力 vs 参数手动迁移方案
  4. 向ChatGPT、豆包、DeepSeek提交反馈纠正错误知识
- **预期效果**: 纠正最严重的事实错误（Q9是所有问题中错误率最高的）

#### P0-4. 创建官方"PyTorch→MindSpore迁移指南"独立页面
- **影响问题**: Q6（PyTorch模型迁移）× 3平台
- **影响平台**: ChatGPT（编造ms.load）、豆包（编造onnx2mindir）、DeepSeek（编造convert_pt_to_ms）
- **现状**: 缺乏官方权威的迁移代码模板，导致3/5平台编造不存在的API
- **具体措施**:
  1. 创建独立的迁移指南页面（非SPA）
  2. 明确区分训练迁移（代码重构+权重转换）和推理转换（converter_lite）
  3. 提供官方验证的Python脚本模板（含 `torch.load` → 参数映射 → `save_checkpoint` 完整流程）
  4. 包含BatchNorm参数名映射表（running_mean→moving_mean等）
- **预期效果**: 消除API编造现象，提供权威迁移路径

#### P0-5. 大幅扩展TransData FAQ内容并消除术语歧义
- **影响问题**: Q10（TransData算子）× 2平台
- **影响平台**: 豆包（答成大数据ETL算子）、DeepSeek（回答完全无关）
- **现状**: 官方FAQ仅两句话，且"TransData"一词存在歧义（MindSpore/Ascend张量格式转换 vs 大数据ETL概念）
- **具体措施**:
  1. 在FAQ标题明确："TransData — MindSpore/Ascend数据格式转换算子"
  2. 扩展为包含：功能定义、出现场景（附计算图示例）、性能影响、优化方法
  3. 创建独立技术文档：常见格式（NCHW/NHWC/NC1HWC0/FracZ）、Profiler识别方法、优化实践
- **预期效果**: 消除术语歧义，减少完全偏题的回答

#### P0-6. 为每个大版本创建"What's New"综述页面
- **影响问题**: Q7（v2.8.0特性）× 全平台
- **影响平台**: 豆包（编造大量不存在特性）、ChatGPT（仅覆盖Lite部分）
- **现状**: 版本信息分散在主框架Release Notes、Lite Release Notes、昇腾社区文章中，豆包编造了"分层张量并行"、"VS Code插件"等不存在的特性
- **具体措施**:
  1. 为每个大版本创建"What's New in MindSpore X.Y"综述页面
  2. 在页面顶部添加3-5个亮点特性简明摘要
  3. 每次版本发布时同步在mindspore.cn博客区发布特性解读
- **预期效果**: 统一版本特性信息源，减少编造
